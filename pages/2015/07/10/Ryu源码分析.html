Ryu源码分析

Date 2015-07-10 Tags RYU souce code

1 Event类

位置：ryu\lib\alert.py中Event类

有很多属性 调用类方法parser返回一个名为msg的Event类？和ev.msg什么关系？

2 讯息实体msg的类：MsgBase()类（讯息实体的基本类）

位置：ryu\ofproto\ofproto_parser中 MsgBase()类，

构造函数后有属性：datapath，version，msg_type，msg_len，xid，buf 等，属于所有讯 息实体msg的基本属性。此类会被其他产生讯息的类继承。同时继承这些基本属性。

3 datapath

类位置：ryu\controller\controller.py中

Datapath(ofproto_protocol.ProtocolDesc) 可以看出datapath继承了ProtocolDesc类（见下条，继承的类中得到datapath的两个属性）

属性:id，ports，ofproto,ofproto_parser（继承来的）等

方法：send_msg，还有send_packet_out，send_flow_mod等，其实内部最后都调用了 send_msg()

4 datapath的两个属性ofproto,ofproto_parser

dp = ofproto_protocol.ProtocolDesc(version=ofproto_v1_3.OFP_VERSION)

位置：\ryu\ofproto\ofproto_protocol.py中定义了类ProtocolDesc

解释：通过ProtocolDesc类构造函数（init）获得属于datapath的两个属性ofproto,ofproto_parser(即相应版本的协议模块名和协议解析模块名字，比如模块ofproto_v1_3, ofproto_v1_3_parser)，则这个模块成为data path的属性，通过dp.ofproto_parser可以调用相应版本的类、函数等。

5 传统数据包解析

传统包头协议位置：\ryu\lib\packet 中有ethernet ipv4等模块，通过Packet编解码传统包。 用法：pkt = packet.Packet(msg.data) 通过Packet类解析出传统数据包

eth=pkt.get_protocols(ethernet.ethernet)[0] 通过ethernet.py中ethernet类匹配出帧头 dst = eth.dst src = eth.src

帧头属性（如dst , src查看ethernet类即可）

6 ofproto包：

位置：\ryu\ofproto

交互的包形成过程：

  • 交换机发往控制器：

    协议指定 -> 讯息产生,讯息自动解析(框架自动完成) -> 产生讯息实体msg（ev.msg） ->讯息实体msg解析(通过msg属性，如msg.match得到匹配部分)

  • 控制器发往交换机： 协议指定 ->讯息实体msg反解析(形成match和actions等)->产生讯息实体msg(如OFPPacketOut) –>产生讯息(send_msg())

注意：本节注意讯息和讯息实体（msg）概念的区分

6.1 ofproto_protocol.py模块（协议指定）

作用：协议指定

过程：通过里面的ProtocolDesc()类来初始化data path的两个属性(如初始化出ofproto_v1_3, ofproto_v1_3_parser)，以此完成OF协议版本的指定

6.2 ofproto_parser.py模块（产生讯息后，对讯息解析，产生msg讯息实体）

作用：针对交换机发往控制器讯息的解析，为了之后得到讯息实体（msg），msg即控制器和交换机之间 openflow包(除去包头)的负载部分

过程：针对交换机发往控制器讯息的解析，框架自动调用header()和msg()方法，依次解析讯息（根据OF协议版本）。

除此，里面有MsgBase()类，将被协议解析模块中的类继承，以此获得讯息实体(msg)。

6.3 of协议解析模块（如：ofproto_v1_3_parser.py）

类比较多，分类别讨论：

这里，主要讨论继承类MsgBase()的Openflow讯息类,这些类产生交互的讯息实体msg

作用：根据控制器和交换机交换交互的包，用模块中相应类产生出讯息实体msg（of负载部分）。这种类分两种：第二种情况，msg实体在类中得到相应属性，并继承MsgBase()一些属性,而第一种情况需要主动调用类，写入属性值。

  • 第一种，交换机发往控制器消息：

    举例：控制器收到的PacketIn包的类定义：

    @_register_parser
@_set_msg_type(ofproto.OFPT_PACKET_IN)
class OFPPacketIn(MsgBase):
def __init__(self, data path, buffer_id=None, total_len=None, reason=None,table_id=None, cookie=None, match=None, data=None):

    分析：对于packin包，通过这个类产生相应讯息实体msg，msg有两种属性：

    • 特定Msg属性：对于OFPPacketIn的msg，里面有datapath, buffer_id,total_len,reason,tab le_id,cookie,match,data属性，其中data即传统数据包的以太帧，match是OFPMatch的对象实例，实例中有一个40元组的字典型属性（暂且这么理解），键比如in_port，通过match[in_port]访问匹配域的入端口，对象match竟然可以这么访问，OFPMatch中定义的吧）
    • 基本属性：因为都继承类MsgBase(),则也继承了类MsgBase()中属性，作为msg基本属性： datapath，version，msg_type，msg_len，xid，buf

    原理：

    类OFPPacketIn是交换机发给控制器讯息的类，对于交换机发给控制器的讯息，会触发此类,但由于此类用修饰符@_register_parse注册，则此类生成实体前，会先在修饰符函数_register_parse中调用此类中的parser函数进行对讯息进行解析，完成此类的msg实例化，产生讯息实体msg。

    使用方法：如处理packin消息：

    @set_ev_cls(ofp_event.EventOFPPacketIn, MAIN_DISPATCHER)
def packet_in_handler(self, ev):
msg = ev.msg   得到msg实体
…

    解释：以上原理中所说的（Of讯息产生，并解析出msg讯息实体）相当于产生一个对应事件ev，应自定义事件管理函数（如packet_in_handler）对事件进行处理。

    事件管理函数(Event hander)是以接收的事件ev为形参，使用@set_ev_cls（`ryu.controller.handler.set_ev_cls）修饰的函数。修饰符函数@set_ev_cls中第一个参数接受的讯息（Event+讯息类名），第二个参数有四种情况，描述交换机状态。（ryu.controller.handler中定义）。

    其中第一个参数定义见ryu.controller.ofp_event.py，其中函数_create_ofp_msg_ev_class(msg_cls) 调用函数_ofp_msg_name_to_ev_name(msg_cls)产生事件名字作为set_ev_cls参数。

    def _ofp_msg_name_to_ev_name(msg_name):
return 'Event' + msg_name

    其中ev.msg是用来储存对应事件的OpenFlow 讯息实体。上例中是ryu.ofproto.ofproto_v1_3_parser. OFPPacketIn。

    msg.datapath是存储OpenFlow交换器ryu.controller.controller.Datapath类所对应的实体。

  • 第二种，控制器发往交换机消息

    作用：这些类无@_register_parse的类（如OFPPacketOut），这些类用来控制器下发讯息产生讯息实体msg，方法是手动书写调用此类，参数是类应有的属性值如datapath，和情况a过程相反。产生讯息实体msg后通过msg的方法msg_send（msg）产生讯息发出去。

    用法（根据类定义）：如类OFPPacketOut用法：

    def send_packet_out(self, datapath, buffer_id, in_port):  #自定义
    ofp = datapath.ofproto
    ofp_parser = datapath.ofproto_parser
    actions = [ofp_parser.OFPActionOutput(ofp.OFPP_FLOOD, 0)]
    req=ofp_parser.OFPPacketOut(datapath, buffer_id,in_port, actions) #调用类形成讯息实体msg
    datapath.send_msg(req)  #使用datapath方法把msg发出去

对于第一种情况（交换机发往控制器）的几种类：

  • OFPPacketIn(MsgBase):
  • OFPFlowRemoved(MsgBase):当交换机流表项被删除或是过期，向控制器发此消息
  • OFPPortStatus(MsgBase)：当交换机端口状态改变，向控制器发此消息
  • OFPErrorMsg(MsgBase): 交换机发往控制器关于发生错误的消息
  • OFPGetConfigReply(MsgBase)：对于OFPGetConfigRequest的回答消息
  • 对于b情况（控制器下发给交换机）有类：
  • OFPFlowMod(MsgBase)：控制器向交换机发改变流表项的消息
  • OFPPacketOut(MsgBase)：控制器向交换机发的消息
  • OFPGetConfigRequest(MsgBase)：控制器发往交换机的交换机配置信息请求信息
  • OFPSetConfig(MsgBase): 控制器下发的交换机配置设置的要求消息，无需reply

Ryu框架自动调用完成的类：OFPHello，OFPEchoReply，OFPEchoRequest，OFPSwitchFeatures（回答控制器发往交换机OFPFeaturesRequest，前面四个是交换机发往控制器的）

注意：此类还有很多，只要是继承MsgBase的类，都属于产生控制器交换机交互讯息的类。

6.4 讯息实体msg解析

作用：解析讯息实体msg，获得讯息实体内容。

对于控制器解析出的msg讯息实体，一般会有一些属性datapath，match，data等，具体根据哪种msg决定

1 一般属性的解析，如msg属性buffer_id，datapath可直接访问，如msg.datapath

2 对于match这个属性解析和反解析（生成）：

主要类：OFPMatch(StringifyMixin)类，形成match实体。 OFPMatch类生成对象实例match，对象match中一个40元组的字典型属性（暂且这么理解，则也可把match对象看做字典型，和字典一样操作）,字典的键如in_port，ipv4_src等。

  • 解析match：match[键名]获得键值，如msg.match[in_port]得到流表项匹配域的入端口（即数据包入端口）（因为OFPMatch()中定义__getitem__函数，提供下标访问方式，相当于[]重载）

  • 生成match：直接用键值形式作为参数调用OFPMatch() 类：如：

    match = parser.OFPMatch(
in_port=1,
eth_type=0x86dd,
ipv6_src=('2001:db8:bd05:1d2:288a:1fc0:1:10ee','ffff:ffff:ffff:ffff::'),
ipv6_dst='2001:db8:bd05:1d2:288a:1fc0:1:10ee')
if 'ipv6_src' in match:
print match['ipv6_src']

注明：parser是相应版本的解析包名，最好用parser=datapath.ofproto_parser，得到解析包名，如ofproto_v1_3_parser。

当然，还有OFPMatchField匹配域，掩码问题...先不分析。

3 actions

作用：当进入交换机封包满足match匹配则执行动作action(每条流表项可以有多个action，组成action list，因此是数组型)。其次，action要作为指令instruction参数 。

用法：调用action类实例化action。

举例：以OFPActionOutput为例，调用OFPActionOutput类可生成action对象。

OFPActionOutput(OFPAction):用来生成“指定从交换机出口发包”action对象的类（出口动作）。主要用于packout和flowmod的action。

定义：

def __init__(self, port, max_len=ofproto.OFPCML_MAX,type_=None, len_=None):
    super(OFPActionOutput, self).__init__()
    self.port = port
    self.max_len = max_len。

参数：

  • max_len参数针对发给控制器的packin包，当为0时，则不含传统数据包负载；为 OFPCML_NO_BUFFER时，则所有封包加入packin中上传给控制器。缺省值为OFPCML_MAX，表示发给控制器最大包长。

  • Port是包出端口。可以是指定端口号（如1,2等），还可以是： OFPP_IN_PORT，OFPP_TABLE ，OFPP_NORMAL，OFPP_FLOOD， OFPP_ALL，OFPP_CONTROLLER(发给控制器),OFPP_LOCAL，OFPP_ANY具体含义见定义（如ofproto_v1_3.py中）。

其他类：除了OFPActionOutput，还有OFPActionGroup(OFPAction) OFPActionSetQueue(OFPAction)等类，他们都继承了OFPAction类。用法同OFPActionOutput类生成action对象一样。 注：instruction(指令)

说明：匹配后执行的指令集（数组型，可包含多个指令），指导指令中包含的action如何执行，ryu实现了1.3中7种的6种类型，由四个类实现（其中OFPInstructionActions实现了三种类型）。指令类由action对象作为参数生成instruction对象，instruction作为真正match后执行的动作。

用法：需要调用instruction相关类生成。有四种：OFPInstructionGotoTable，OFPInstructionWriteMetadata，OFPInstructionActions，OFPInstructionMeter，他们都继承OFPInstruction类。

例如OFPInstructionActions：定义如下：

class OFPInstructionActions(OFPInstruction):
    def __init__(self, type_, actions=None, len_=None):
    ...

参数：type_表示匹配后在执行action前的执行动作（不同于action，比如OFPActionOutput表示包出口动作。type_可以是三种：

  • OFPIT_APPLY_ACTIONS(立即执行action list中所有action)；
  • OFPIT_WRITE_ACTIONS(把action list中多个action放入 action set中，等所有流表处理完一 次性处理action set中所有action，只支持单流表无意义)；
  • OFPIT_CLEAR_ACTIONS（清除前面流表所产生action set中所有action，单流表无意义）

actions即为上面的action对象。

因此，OFPInstructionGotoTable（table转移，只支持单流表无意义），OFPInstructionWriteMetadata， OFPInstructionMeter用法同理 其实，目前很多交换机只支持单流表，以上有几个指令是无意义的。

4 data属性

类：class Packet(object):

def init(self, data=None, protocols=None, parse_cls=ethernet.ethernet):

Packet()类用来对传统包编解码（填入包头和解析包头，如ip包头等）

位置：\ryu\lib\packet\packet.py

类方法：add_protocol(self, proto)：如给控制器下发包中传统数据包头添加协议字段（如ip） get_protocols(self, protocol)：解析传统数据包包头协议字段，返回协议对象的列表 get_protocol(self, protocol)：同上，不同是返回与protocol匹配的第一个协议对象 用法：

  • 封包解析

    对于packIn的msg，msg属性data是传统以太帧内容（即进入交换机的封包），但它是bit字符串，需要以data为参数，通过调用类Packet(object)生成对象pkt来使用。

    pkt利用类Packet方法get_protocols()或get_protocol()得到协议字段值（如Ip包头，mac头等）

    例如：

    pkt = packet.Packet(msg.data)    #形成packet对象
pkt_eth = pkt.get_protocols(ethernet.ethernet)[0]    #得到与参数协议匹配的协议字段对象
if not pkt_ethernet:
return   # non Ethernet    
eth_dst=pkt_eth.dst
eth_src=pkt_eth.src

    解释：get_protocols(protocol)中参数是要解析的协议（如ethernet.py中ethernet类匹配出帧头，位于\ryu\lib\packet\中），返回与协议匹配的协议字段对象的列表，因此需要用[0]得到第一个列表中第一个对象。get_protocol(protocol)可以直接返回匹配的第一个对象。

    解析字段pkt_eth属性值选取(如目的地址pkt_eth.dst)参考协议定义（ethernet类的属性）

  • 封包生成

    生成空Packet对象->生成协议对象（ethernet,ipv4等）-> add_protocol()把协议对象依次加入Packet空对象中->Packet对象调用serialize()方法把协议对象序列化成Packet对象的属性data。data封包生成，可以用于packout类的data参数。

    例如：

    pkt = packet.Packet()   #生成空的Packet ()对象
pkt.add_protocol(ethernet.ethernet(ethertype =...,dst=...,src =...))
pkt.add_protocol(ipv4.ipv4(dst=...,src=...,proto =...))
pkt.add_protocol(icmp.icmp(type_ =...,code =...,csum =...,data =...))
pkt.serialize ()   #将添加协议对象序列化为Packet ()对象的data属性
bin_packet = pkt.data  #提取封包好data属性

    另一种写法（用API）：

    e = ethernet.ethernet (...)
i = ipv4.ipv4 (...)
u = udp.udp (...)
pkt = e/i/u
